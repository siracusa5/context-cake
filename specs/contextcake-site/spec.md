# ContextCake Product Site

A public marketing + documentation site for ContextCake: explains the cascade in seconds,
documents the engine properly, and gives visitors a copy-paste install path. Built to feel
like a product, not a GitHub README with a domain.

**Workflow:** Requirements-first (product/greenfield).
**Design doc:** `specs/contextcake-site/design.md` (brand system, architecture, agent handoff).

## Problem Statement

ContextCake has a working engine, playground, and MCP server, but no public face. Today a
prospective user finds a private repo they cannot read, no way to install anything, and no
docs beyond the README. The project reads as a personal experiment rather than a tool a team
could adopt. The site closes that gap: one place that explains *what agents read is a
composite assembled section-by-section across stacked layers*, documents how to run it, and
provides a working install command.

## User Stories

- As an **engineer evaluating ContextCake**, I want the core idea demonstrated visually in
  under 10 seconds, so I can decide whether to keep reading.
- As an **engineer adopting ContextCake**, I want a copy-paste install command and a
  5-minute quickstart, so I can see a cascade resolve on my own machine.
- As an **agent operator**, I want reference docs for the MCP tools and manifest schema, so
  I can wire ContextCake into Claude/other agents without reading source.
- As a **team lead**, I want the trust model stated honestly (manifest is a trust boundary,
  conflicts surfaced not hidden), so I can assess whether to roll it out.
- As the **maintainer**, I want the site deployed through the existing pipeline
  (Cloudflare Pages preview on push, production on GitHub release), so shipping site changes
  costs nothing extra.

## Acceptance Criteria

### Homepage (`/`)

- [ ] WHEN a visitor loads `/` THE SYSTEM SHALL render, above the fold at 375–1440 px wide:
      the value proposition, an install command with a copy button, and CTAs to docs and
      GitHub — with no horizontal scroll.
- [ ] WHEN a visitor activates the copy button THE SYSTEM SHALL copy the install command to
      the clipboard and show visual confirmation.
- [ ] WHEN the hero renders THE SYSTEM SHALL show the cascade resolve visual (animated
      layer-stack → banded composite), with per-band source-layer provenance visible by
      default. *(Phase 5; a static banded composite is the acceptable launch state.)*
- [ ] WHEN `prefers-reduced-motion` is set THE SYSTEM SHALL replace all animation with
      static equivalents.
- [ ] WHEN demo/merge output is shown anywhere on the site THE SYSTEM SHALL render data
      generated by `resolver.mjs` at build time from `playground/demo-layers/` — never
      hand-authored fake output.
- [ ] WHEN the homepage presents features THE SYSTEM SHALL cover at minimum: section-level
      merge, per-section conflicts with dates, provenance, foreign MCP sources, the write
      path, zero-dependency engine, and the MCP tool surface (`search`, `read_file`,
      `list_concepts`, `get_links`).

### Docs (`/docs/*`)

- [ ] WHEN a visitor navigates to `/docs` THE SYSTEM SHALL present a docs shell with
      sidebar navigation, full-text search, and dark/light modes.
- [ ] WHEN the docs are complete THE SYSTEM SHALL contain the four sections defined in
      design.md §5: Getting Started, Concepts, Guides, Reference.
- [ ] WHEN docs describe CLI flags, MCP tools, manifest fields, or override syntax THE
      SYSTEM SHALL match current engine behavior (no references to removed subsystems:
      `--hash`, `--shadow`, `override: exception`, Group layer, recency tiebreak).
- [ ] WHEN a visitor reads the Concepts section THE SYSTEM SHALL state the manifest trust
      boundary plainly (an `mcp` layer spawns a command; only trusted manifests).

### Install (`/install`)

- [ ] WHEN a visitor loads `/install` THE SYSTEM SHALL present copy-paste installation
      steps, prerequisites (Node ≥ 18, nothing else), and a first-run command that produces
      visible output.
- [ ] WHEN the distribution decision (Open Questions) is unresolved THE SYSTEM SHALL NOT
      ship this page publicly with placeholder install commands.

### Site-wide

- [ ] WHEN the site is built THE SYSTEM SHALL produce fully static output deployable to
      Cloudflare Pages with no runtime server.
- [ ] WHEN any page is shared THE SYSTEM SHALL provide OpenGraph title, description, and a
      brand-consistent OG image.
- [ ] WHEN any page renders THE SYSTEM SHALL meet: text contrast ≥ 4.5:1, visible keyboard
      focus states, semantic landmarks, and SVG icons (no emoji as icons).
- [ ] WHEN production assets load THE SYSTEM SHALL serve fonts and scripts self-hosted (no
      CDN/third-party requests).
- [ ] WHEN `npm run build` runs in `site/` THE SYSTEM SHALL exit 0 with zero broken
      internal links.

## Out of Scope

- Building the desktop app / GUI, installers, and auto-update — that product surface is
  owned by `specs/contextcake-distribution/spec.md`. The site documents channels as they
  ship; at site launch the only channel is clone-and-run.
- Hosted/SaaS ContextCake, accounts, telemetry, or analytics requiring consent banners.
- Blog. `/changelog` is generated from GitHub releases; no authored posts at launch.
- Rewriting `docs/architecture.md` — the docs port supersedes it for public consumption;
  reconciling the internal doc is separate work.
- The playground itself (already shipped); the site links to it and screenshots it.

- ~~Distribution~~ **RESOLVED 2026-07-02 (for site launch): git clone only — no npm publish.**
  Deferred pending a supply-chain security review (2026 saw repeated npm compromises of
  AI/agent packages via postinstall droppers — Mastra, TrapDoor; npm v12 changes defaults
  in July 2026). The zero-dependency engine makes clone-and-run the honest path anyway:
  no `npm install`, no install scripts, no transitive deps — the site presents this as a
  security property, not a limitation.
  **The channel roadmap is owned by `specs/contextcake-distribution/spec.md`** (merged in
  2026-07-02): `.mcpb` and Homebrew lead, npm follows a supply-chain-hardening review and
  publishes as `contextcake` (with `context-cake` reserved anti-typosquat). As each channel
  ships, `/install` and the docs installation page gain that channel — the clone-and-run
  story remains the auditability baseline, not a limitation to walk back.
- ~~Domain~~ **RESOLVED 2026-07-02: `contextcake.com`** (registered; only the .com for
  now). Canonical URLs, OG tags, and the Cloudflare Pages custom domain all use it.
  Other TLDs (.dev/.ai/.io) verified available same day but not purchased — revisit
  defensive registration post-launch.

**Resolution gate:** repo visibility is now the single remaining gate. Install content is
written against the public GitHub URL and clone-based install — it cannot go live while
the repo is private.

## Dependencies

- **Engine repo** — docs content sources: `README.md`, `playground/README.md`, `CLAUDE.md`,
  `specs/contextcake-core/design.md`.
- **Demo-data seam** — a small wrapper enumerating concepts and calling `resolveConcept`
  per concept (the resolver handles one concept at a time; no all-concepts dump exists).
  Build-time only; lives in `site/scripts/`.
- **Hero visual seed** — reviewed prototype vendored at
  `specs/contextcake-site/assets/cascade-viz-prototype.html` (banded composite +
  persona selector already validated; cake→column collapse animation not yet built).
  Build guidance: design.md §11.5.
- **Deploy pipeline** — Cloudflare Pages; preview via push to main (`/deploy-preview`),
  production via GitHub release (`/go-live`).
- **Distribution roadmap** — `specs/contextcake-distribution/spec.md` owns install
  channels, the GUI, and auto-update; `/install` grows a section per channel as they ship.
